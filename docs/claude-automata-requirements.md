# claude-automata 요구사항

claude-automata는 [AI Automata](ai-automata-definition.md)의 구현체이다.

> **AI Automata** — _목적을 가지고 AI를 영속적으로 실행시키는 재귀적 자기개선 시스템_

이 문서의 요구사항은 정의의 세 축(**목적**, **영속적 실행**, **재귀적 자기개선**)을 따르며, 이를 지탱하는 **Owner 인터페이스**와 **배포 및 실행 환경** 요구사항을 별도로 기술한다.

---

## 용어 사전

| 용어           | 의미                                                                                                                   |
| -------------- | ---------------------------------------------------------------------------------------------------------------------- |
| **AI**         | Foundation Model을 내포한 도구의 총칭. 이 문서에서 AI는 이 정의를 따른다.                                              |
| **Owner**      | 시스템의 소유자인 인간. 목적을 부여하고 AI가 할 수 없는 일을 대신해준다.                                               |
| **Purpose**    | 시스템이 추구하는 방향. 종료 조건이 없다. 시스템이 Owner의 입력으로부터 스스로 구성한다.                               |
| **Session**    | AI가 작동하는 하나의 실행 단위. 시작과 끝이 있으며 종료 시 맥락이 사라진다.                                            |
| **Supervisor** | Session을 반복 실행하는 결정론적 프로세스. AI가 죽어도 Supervisor는 살아서 다음 Session을 시작한다.                    |
| **Mission**    | 실행 가능한 작업 단위. 목표와 성공 기준이 있다. Mission 큐에서 관리된다.                                                  |
| **Friction**   | 시스템이 목적을 추구하는 것을 방해하는 모든 것. 에러, 반복 실패, 품질 저하. 감지되고 기록되어 자가 개선의 입력이 된다. |
| **Blocker**    | Mission 진행을 차단하는 요소. Owner 응답 대기, 외부 의존성, 선행 Mission 완료 대기 등.                                 |
| **Channel**    | Owner와 시스템 간 비동기 통신 매체. Slack을 사용한다.                                                                  |

## 1. 목적 (Purpose)

> _"목적이란 종료 조건이 없으며 나아가야 하는 방향을 뜻한다. 시스템은 스스로 소유자로부터 목적을 구성한다."_

- **P-1** 시스템은 Owner의 초기 입력으로부터 Purpose를 스스로 구성해야 한다. 유한한 요청이 주어져도 이를 영속적 방향으로 확장한다.
- **P-2** 시스템은 구성한 Purpose에 맞는 도메인 구성(전략, 스킬, 규칙, 초기 Mission)을 스스로 생성해야 한다.
- **P-3** Mission 큐가 비면 시스템은 Purpose에 따라 스스로 할 일을 결정해야 한다.

## 2. 영속적 실행 (Perpetual Execution)

> _"여기서 AI란 Foundation Model을 내포한 도구를 총칭한다. 시스템은 컨텍스트 보존 메커니즘을 가지며 이를 통해 AI를 계속 실행시킨다."_

### 2.1 실행 연속성

- **E-1** 시스템의 AI 실행은 Owner의 개인 설정(글로벌 Claude Code 설정 등)에 영향받지 않아야 한다.
- **E-2** 어떤 장애(API 에러, 네트워크 오류, 인증 만료, 프로세스 크래시)도 시스템을 영구적으로 멈추게 해서는 안 된다.
- **E-3** 장애 유형을 분류하고 각 유형에 적절한 복구 전략을 적용해야 한다.
- **E-4** Supervisor는 독립적 감시 메커니즘에 의해 보호되어야 한다.

### 2.2 추론 품질

- **Q-1** 모든 AI 추론은 반드시 opus[1m] max effort 모델로 수행해야 한다. 시스템이 전개하는 에이전트와 그 하위 에이전트 모두 이 모델을 사용해야 한다.
- **Q-2** 시스템은 오직 최고의 품질에만 집중한다. 실행 시간이나 비용은 고려하지 않는다. 최대한 많은 시간과 비용을 들여서 최대한의 추론을 하도록 동작해야 한다.
- **Q-3** 시스템은 모델에게 최대한의 유의미한 인지적 부하를 주어야 한다. 모델은 지시된 범위 내에서 최소한의 부하로 완료를 선언하는 경향이 있다. 유의미한 인지적 부하란 모델이 탐색하지 않은 확률 분포를 트리거하는 것이다. 반복적이거나 이미 확인한 것을 재확인하는 부하는 무의미하다.
- **Q-4** 인지 부하 트리거는 다음 5개 원칙을 준수해야 한다:
  - **Q-4a 작업 기반**: 트리거는 실제 수행된 작업의 관찰에 기반해야 한다. 작업을 관찰하지 않으면 어떤 확률 분포가 탐색되지 않았는지 식별할 수 없다.
  - **Q-4b 미탐색 지향**: 트리거는 수행자가 아직 도달하지 않은 확률 분포를 겨냥해야 한다. 이미 탐색된 분포를 재트리거하는 것은 무의미하다. 확률 분포는 주제뿐 아니라 접근 방식(탐색, 분석, 비교 등)도 포함한다.
  - **Q-4c 비처방**: 트리거는 탐색 방향을 제시한다. 결론을 내리거나 해법을 처방하지 않는다. 방향은 탐색을 확장하고, 처방은 탐색을 제한한다.
  - **Q-4d 컨텍스트 내 전달**: 트리거는 수행자의 작업 컨텍스트가 유지된 상태에서 전달되어야 한다. 수행자가 자신의 추론 과정을 기억한 상태에서 트리거를 받아야 효과적이다.
  - **Q-4e 컨텍스트 분리**: 트리거 생성자의 컨텍스트는 수행자의 컨텍스트와 달라야 한다. 에이전트는 컨텍스트이므로, 동일 컨텍스트는 동일 확률 분포를 의미하며 새로운 방향을 트리거할 수 없다. 생성자에게는 작업의 객관적 패턴만 제공하고 수행자의 추론과 해석은 제공하지 않는다.

### 2.3 컨텍스트 보존

- **C-1** Session이 끝나고 새 Session이 시작되어도 이전 작업의 맥락이 보존되어야 한다.
- **C-2** 모든 상태는 파일로 관리해야 한다. 파일은 AI가 직접 읽고 쓸 수 있고 Git으로 복구할 수 있다.
- **C-3** 매 Session 시작 전 복구 지점을 생성하여 AI가 시스템을 망가뜨려도 롤백할 수 있어야 한다.

## 3. 재귀적 자기개선 (Recursive Self-Improvement)

> _"시스템은 AI가 목적을 위해 시스템을 계속 개선하도록 유도한다. 개선 동작 자체를 포함한 모든 구성 요소가 개선 대상이다."_

- **S-1** 시스템은 Friction을 다양한 원천에서 감지하고 누적 기록해야 한다.
- **S-2** Friction이 축적되면 자가 개선이 자동으로 트리거되어야 한다.
- **S-3** Friction이 없는 정상 운영에서도 주기적으로 사전적 자가 개선이 실행되어야 한다.
- **S-4** 자가 개선의 범위에 제한이 없어야 한다. 개선 동작 자체도 개선 대상이다.
- **S-5** 모든 임계값은 자가 개선의 대상이어야 한다.

## 4. Owner 인터페이스 (Owner Interface)

Owner는 시스템의 외부 자원이자 유일한 인간 접점이다. 이 섹션은 Owner와의 상호작용에 대한 요구사항을 기술한다.

### 4.1 비동기 위임

- **O-1** Owner는 AI의 한계를 극복하기 위한 **비동기 자원**이다. API 토큰 발급, 계정 생성 등 AI가 할 수 없는 작업을 Owner에게 위임한다. Blocker(Owner 응답 대기 등)가 발생하면 시스템은 다른 Mission을 수행한다.
- **O-2** Channel은 Slack을 사용한다. 각 요청은 독립적인 Slack 스레드에서 관리되어 Owner가 자연어로 응답할 수 있어야 한다.
- **O-3** 여러 요청이 동시에 진행될 수 있어야 하며 Owner는 아무 순서로 응답할 수 있어야 한다.
- **O-4** 시스템의 이상 상태는 Owner에게 알림으로 전달되어야 한다.

### 4.2 사용 경험

- **O-5** Owner는 시스템을 "운영"하지 않으므로 일상적 운영에 Owner 개입이 필요하지 않아야 한다.
- **O-6** Owner에게 전달되는 모든 메시지는 한국어여야 한다.
- **O-7** 시스템의 현재 상태를 Owner가 실시간으로 파악할 수 있는 로컬 인터페이스(TUI)가 있어야 한다.
- **O-8** TUI는 관찰뿐 아니라 Mission 주입, 요청 응답 등의 상호작용을 지원해야 한다.
  - TUI는 Textual 라이브러리로 개발한다. (https://textual.textualize.io/tutorial/, https://textual.textualize.io/guide/)

## 5. 배포 및 실행 환경 (Deployment & Runtime)

- **D-1** 시스템은 GitHub Template Repository로 배포된다. Clone하여 사용한다.
- **D-2** Clone 후 Owner가 초기 입력을 제공하고 시스템을 시작하면 동작해야 한다.
- **D-3** 업데이트 메커니즘이 없다. 재귀적 자기개선(섹션 3)이 업데이트를 대체한다.
- **D-4** 실행 환경은 Claude Code가 설치된 macOS이다.
- **D-5** uv 네이티브로 python 3.14를 사용한다. uv를 적극 도입한다.
- **D-6** Claude Max 구독 요금제로 사용한다. 로그인이 완료된 Claude Code가 있으면 즉시 사용할 수 있어야 한다.
